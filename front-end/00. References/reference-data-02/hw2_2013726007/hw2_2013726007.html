1.웹서비스 설계


과제 #1의 기획을 보완, 확장, 구체화

-> 초기 기획 당시 내가 생각했던 키워드는 간단, 신속이였다, 그 컨셉은 요번 과제에도 크게 달라지지 않았다 여전히 누구나 사용 할 수 있도록 심플하게 인터페이스를 구성하려고 노력했고 간단한 검색 조건을 통하여 빠르게 사용자가 검색 결과를 얻어 낼 수 있도록 설계하고자 했다 결과적으로 과제1에서 설계했던 스토리보드와 요번에 1차적으로 설계한 인터페이스는 크게 다르지 않았다.


다른점이 있다면 검색조건을 다 달아놓고, 찾기 버튼을 눌렀을 때 새로운 창에서 나오도록 계획을 했었지만 사용자가 결과를 보면서 검색조건을 바꾸는 경우도 충분히 생길 수 있다고 생각하여 변경을 했다, 또한 사용자가 선택한 약국만 마커표시 버튼을 누르면 마커가 지도에 생성되도록 할 예정이였지만, 이것 또한 사용자가 주소만 보면서 비교를 하기엔 무리가 있다고 판단하여 찾기 버튼을 누르면 바로 지도에 마커가 나오도록 설계를 하였다. 또한 웹사이트 화면 전체를 사용하는것이 아니라 가운데 반정도만 사용함으로써 좀더 사용자가 가독성있게 서비스를 이용 할 수 있도록 계획했다.


상호작용 방식 및 흐름, 서비스 구성

-> 일단 검색조건은 사용자가 텍스트박스에서 입력해야되는부분과 샐렉트박스와 체크박스로 체크해야되는 3가지 부분으로 나눈다 이 부분은 사용자 대상 실험을 통해서 충분히 달라질수있는 부분이지만, 현재는 위와같이 구현을 하였다. 먼저 지역을 입력해준다 이 부분을 샐렉트박스를 통해서 할지 입력을 할지 고민을 많이 했지만 샐렉트박스로 고르는 것보다 사용자가 직접 입력을 하는것이 빠를것이라 판단하여 현재는 텍스트박스로 구현을 하였다, 혹시나 사용자가 양식에 맞지 않게 입력을 할까봐 placeholder를 이용해 예시를 보여준다,


시간을 고르는 셀렉트박스는 2시간 단위로 구역을 나눴고, '아무 선택을 안할때는 사용자가 검색을 하는 시간에 해당하는 부분에 선택이 되있도록 구현을 하였다'. 사용자로 정한 주 타겟이 빨간날의 약국을 급하게 이용해야되는 사람들이기 때문에, 평일을 선택에서 제외하고 체크 박스로 주말과 공휴일을 선택하도록 하였다, 이 부분을 시간처럼 샐렉트 박스를 이용해서 월~일,공휴일로 나누어 똑같이 사용자가 검색하고있는 날을 default 선택값으로 하는 방법을 고민하고 있는데, 사용자 실험을 통해 두 방법을 비교해보고 결정할 생각이다.


검색 조건을 다 수행하고 찾기 버튼을 누르면 검색 결과 부분에서 파씽된 데이터(약국이름과 주소)와 구글지도에서 해당하는 주소에 마커가 생성된다, 사용자가 지도에 있는 마커를 보면서 사용자가 마커를 클릭하면 말풍선이 뜨면서 주소와 이름이 나오도록 구현을 하였고 아직 구현을 못했지만 마커를 누르면 해당 약국에 간략한 정보(이용시간, 전화번호)등이 옆에 텍스트박스에 뜰수있게 구현을 할 계획이다 또한 프로토타입을 만들고 사용하면서 초기화를 해야되는 경우가 굉장히 많은데 f5를 눌러서 초기화를 하는 방법이 있지만 사용자가 그 방법을 모를 수 있는 경우를 생각하여 초기화메뉴를 만들어 초기화를 할 수 있게 했다

공공데이터 활용방법

1. 공공 데이터를 받아올때 요청메세지로 요일, 지역(시,구), 진료시간등을 사용자가 입력한 검색조건에 맞는 변수를 통해 받아오면 파씽을 한다


2.위에서 파씽된 주소 데이터를 string 부분 비교를 통해 구체적인 주소 예를 들어 대화동,일산서구등과 비교를 하여 파씽된 데이터를 거른다, 이 과정이 필요한 이유는 파씽해서 데이터를 가져올때 지역을 시단위로 밖에 받아 올 수 없기때문이다.


3.위도,경도를 이용해 마커를 생성하고, 그밖에 정보등을 이용해 마커를 클릭하면 나오는 말풍선을 채워준다


2.대화형 프로토타입


2.1 대화형 프로토타입 링크, 문제점 및 해결방안


약국 찾기 프로토타입

링크 타고 들어가기

계획단계에서 구현하지 못했던 문제였던 구체적인 주소를 입력하면 입력한 string과 출력메세지로 공공api에서 받은 데이터를 부분비교를 하여 한번 걸러서 출력 해주는 기능을 추가하였다 어디서 조건문을 걸어줘 걸러야 할까 고민을 많이 했었는데 파씽을 해서 데이터를 받는 response함수에서 조건문을 추가해줘 해결하였다.


파씽되어 데이터를 받을 때 호출메세지로 데이터를 20개만 받는걸로 한정 지었다, 하지만서비스가 구체화 되면서 제약조건들이 많아지면서 최종적으로 나오는 결과의 수가 너무 적은 문제점이 있어 100개로 늘렸다.


프로토타입으로 사용자 분석을 하던중 월~금을 묶어서 평일로 선택하게 했었는데 파씽된 데이터를 보니깐 평일에도 요일마다 영업시간이 다른 약국들이 많아서 요일을 좀더 월~일,공휴일로 세부적으로 나누어 검색조건을 강화했다.


또한 검색을 하고 재검색을 할경우 마커가 생성되지 않는 문제점이 발생하였다. 코드에서 동적으로 여러개의 마커와 말풍선을 포문으로 생성하는데 여러 배열들의 인덱스를 제대로 초기화 해주지못해서 문제가 발생해 해결을 하였다.


2.2 세부 구현 방식

'html'에서는 bootstrap을 이용하였다, 이용한 컴포넌트로는 앞에서도 얘기 했지만 셀렉트박스와 inline-input textbox, 버튼을 이용하였고 사용자가 좀더 집중력있게 사용을 하도록 점프트론을 이용하여 입력부분과 결과 출력부분을 나누어 디자인을 했다


'css'에서 가장 고심했던 것은 점프트론과 컨테이너의 간격이다 너무 좁거나 넓으면 안에 있는 컴포넌트들이 난잡해보이거나 한눈에 안들어오는 경우가 생기기 때문에 어떡하면 컴포넌트들이 '가시성'있게 만들어줄지 고민을 많이했다. 또한 각종 컴포넌트들과 테그들의 색을 통일해줘 단순해보이고 심플하게 보이도록 설계하였다.


'자바스크립트' 부분에서는 jquery를 이용하여 입력부분에서 입력받는 것을 변수로 받아 그것을 이용하여 xml파씽을 할 때 호출메세지 변수로써 사용하였고, 좀더 데이터를 구체적으로 받아오기 위하여 구제적인 주소를 받아 그것을 string으로 비교를하여 한번 필터를 했습니다, 또한 셀렉트박스 시간과 요일을 고르는 부분은 사용자들의 편의를 위하여, default값으로 앞서 설명했듯이 현재 사용자가 검색하는 그때의 시간을 변수로 받아서 그때의 시간,요일에 해당하는 셀렉트요소로 자동으로 선택이 되도록 하였다, 또한 호출이 성공하여 데이터를 파씽하는 부분에서 위도와 경도를 배열로 저정하여 구글맵에서 마커를 생성하고, 첫번째 위도와 경도를 map의 center로 바꿔줘 사용자들이 빠르게 지도를 탐색할수있도록 하였다. makemaker 함수는 마커를 생성하고 말풍선 또한 생성해준다.


2.3 디자인

전체적인 디자인은 '휴일지킴이 약국사이트'를 '메타포'로써 인터페이스를 참고했다, 메타포 디자인 지침에 따라 핵심적인 요소인 빨간날 영업하는 약국을 검색하는 기능과 지역을 검색하여 약국을 찾아주는 기능을 뽑았고, 날짜까지 검색조건에 추가한 부분을 삭제하고 요일로만 검색이 가능하도록 축소화시켰다. 또한 사용자들이 이용하기 유용하도록 자동적으로 현재 검색하고 있는 시간과 요일을 default값으로 하여 사용자들이 지역만 입력해도 무방하도록 디자인했다. 전체적인 구도 디자인은 컨테이너를 이용하여 웹페이지를 좁게 사용한다 앞서 계속 강조 했듯이 간단하고 빠르게 검색이 가능하도록 하는것이 목표 이기 때문에 몇개없는 컴포넌트들이 퍼져있는것이 아니라 좁게 배치하여 한눈에 들어오도록 하였다. 또한 색깔을 포인트로 준 점프트론을 배치하여 좀더 '가시화'를 강조하여 기능들이 한눈에 들어오도록 하고 행위유도성을 강화하여 빠르게 검색을 할수있도록 유도하였다.


또한 휴일 지킴이에는 없는 기능인 지도 마커 기능은 네이버, 다음등에서 장소를 검색할때 나오는 지도api를 메타포로써 착안했다. 예를 들어 광운대 화장실하면 나오는 마커와 마커를 누르면 나오는 정보창등을 약국찾기에서도 그대로 사용하였다. 다른점은 마커를 누르면 나오는 말풍선에서 제공하는 정보는 간단하게만 제공하고 지도 옆에 textarea에서 나머지 자세한 정보를 볼수있게하여, 지도에서는 간단한 정보들을 확인하면서 위치를 기반으로 찾는거에 집중하도록 제약조건을 달아줬다.

3.사용자 대상 실험


3.1 계획수립

실험 목표 -

다른 검색 방법과 다르게 얼마나 '정확'하고 '신속'하게 검색이 되었는지를 평가할 것 이다, 정확은 사용자의 설문조사로 평가를 하고, 신속은 시간을 재서 평가를 할 것이다.

실험 과정 -

제대로 검색이 되었는지 정확도를 측정하기 위하여 평소에 자신이 잘아는 지역을 검색하도록 주문 하고 같은 검색 질의를 네이버를 이용하는 방법, 휴일 약국지킴이를 이용하는 방법, 대화형 프로토타입을 이용하는 방법 3가지로 나눠 순차적으로 검색을 해보도록 할것이다. 만약 사용자가 검색을 하는데 해매거나 40초를 초과한다면 다음 방법으로 넘어가도록 유도를 할 것 이다. 모든 검색이 끝나면 설문조사와 인터뷰를 통하여 얼마나 정확했는지, 각각의 방법 의 장점과 단점 그리고 대화형 프로토타입의 고칠점등을 설문 조사를 통해 조사 할 것이다.

인터뷰/설문 문항 마련-

설문조사와 인터뷰를 설계하면서 중점을 둔점은 정성적 평가이다, 정확도는 어찌보면 정량적 평가일 수도 있지만 실험 설계를 할 때 정확도에 대해 정확한 기준을 정하지 않았고 설계를 할 때 사용자에게 자신이 잘아는 지역안에서 찾아보라는 조건을 걸어 사용자가 실험을 진행하면서 자신만의 기준으로 얼마나 정확한지 판단하기 때문에 정성적평가라고 간주하고 실험과 설문조사를 설계 하였다 처음 인터뷰 내용인 각각의 장점과 단점을 조사하는 부분은 사용자가 원하는 정보를 얼마나 제약조건과 행위유도성을 고려하여 원하는 정보를 제공했는지에 대해 알아보기위한 질문이다 이 질문에서 신속함과 정확도를 알 수 있는것은 물론이고 메타포를 잘 사용하여 메타포의 대상보다 내가 원하는 방향으로, 발전된 방향으로 설계가 되었는지 확인할 수 있다. 설문조사 부분에서는 정성적인 정확도를 어느정도 수치화를 통해 사용자가 왜 그렇게 정확도를 책정했는지 물어봄으로써 사용자가 생각하는 정보의 정확도를 파악 할 수 있는 척도를 얻을 것이다

3.2 실험 준비

실험 환경은 장소에 구애를 받지 않아야하는 서비스이므로 장소를 따로 정하지는 않았다, 사전에 여러 실험조건을 시뮬레이션 해봄으로써, 자신이 잘아는 지역으로 해야 데이터 정확도 평가를 할수있겠구나 알 수 있었다. 약식으로 참여한 참여자들 5명, 정식으로 참여한 참여자들 3명이였다. 이와 별개로 프로토타입을 구현할때 많은 사람들이 사용을 해보게 하고 피드백을 얻었다.

3.4 분석 및 결론

전체적 요약-

전체적을 잘 진행 된 부분은 사용자들이 시나리오대로 잘 따라와준것이 가장 잘 진행된 부분이다, 설계 할 때 잡았던 컨셉인 빠른 검색이 잘 이루어졌다. 도표를 보면 3가지 방법에 비해 압도적으로 빨랐다, 정확도 부분에서도 높진 않지만 빠른 검색을 위해 몇몇 부분을 포기한것 치고는 꽤 높은 정확도를 얻어 만족한다. 또한 인터뷰와 설문조사를 통해서도 사용자들이 주의가 분산되지 않고 컴포넌트들이 잘 배치 된것 같다고 평가를 해주었는데 이 부분 역시 디자인할 때 많은 고민을 한부분인데 설계대로 잘 이루어진 것 같다.


오히려 많은 데이터들이 사용자들의 정확도를 떨어트릴꺼라 생각하고 자료에서 뜨는 데이터의 양을 제한 했었는데 많은 사용자들이 네이버,다음의 비해 데이터양이 부족하다 라고 말해줘 사용자들이 생각하는 정확도는 무언인지 알수있는 기회가 되었다, 또한 사용자들에 input-text에 있는 placeholder를 보고 양식에 맞게 지역을 검색 해줄거라 생각을 했는데 사용자들이 그 예시를 보지 않고 자기 멋대로 검색을 하는경우가 있었다. 또한 검색을 하는데 약간의 딜레이가 생기면 제출을 작동이 안된다고 생각하여 초기화를 누르는 사용자도 있어 많이 당황했었다. 사용자들이 요구한 기능으로는 내 위치가 마커에 뜨는 기능을 원했고 검색이 성공했거나 실패했을때 확실한 피드백을 원했다. 결과창이 왼쪽 텍스트 박스의 좀더 네이버나 다음 지도에서 마커를 눌렀을 때 처럼 더 다양한 정보와 눈에 확들어오게 디자인을 요구했다.

문제점 원인 분석-

일단 가장 큰 원인은 '나 스스로 사용자라고 착각을 하여 내뜻대로 사용자들이 쓸거라 생각한 점이 본질적인 원인이라 생각한다', 그래서 위에서 말한 문제들이 발생한 것이다, 지도에 너무 많은 마커들이 생성되면 사용자들이 고르는것이 힘들까봐 데이터 양을 한정지었었는데, 오히려 사용자들은 많은 데이터들을 보면서 고르는것을 원했고 직접 입력하는것이 속도가 더 빠르다 생각하여 셀렉트박스를 안쓰고 input-text로 주소를 입력받은것이 검색이 익숙하지 않은 사용자들에게 훨씬 시간이 오래걸리는 문제가 발생한것이다 또한 약국 같은 경우에는 자신이 잘아는 집 근처나 그나마 잘아는 지역에서 갈줄 알고 내위치 기능을 빼고 마커에만 집중을 했는데 사용자들은 자신의 위치가 나오는 기능을 많이들 원했다.

디자인을 빠른 검색에만 초점을 맞췄고, 네이버나 다음에서 마커를 누르면 제공하는 정보들은 너무 많고 눈에 들어오지도 않았기 때문에 딱 필요한 정보만 제공하기 위해 전화번호와 영업시간만 제공을 했는데 사용자들은 더욱 많은 데이터를 원했다 물론 컨셉을 '심플하고 신속한 검색을 제공하자'이지만 약국찾기는 사용자가에게 원하는 정보를 제공을 하는 서비스이기 때문에 적어도 목적에 맞는 데이터는 제공을 해야된다는 점을 간과 했었던것 같다.

개선 방안 도출-

일단 데이터 양을 조절해주는 방법은 파씽을 할때 요청메세지로 데이터양을 엄청나게 받고 마커를 생성할때 for문으로 30개정도로 한정지어 처리속도와 데이터양을 밸런스있게 잡으려고 한다, 그리고 사용자들이 도,시단위 입력받을 때 샐렉트 박스로 해달라해서 그점은 바로 개선하여 샐렉트 박스로 바꿔 제약조건을 강화했다. 현재 가장 많이 고민을 하고 있는것이 검색 성공이나 실패시 확실한 피드백을 어떻게 보여줄 것인가인데 일단 검색이 실패하면 팝업창이 뜨게하고, 성공하면 앵커를 이용해 화면을 약간 내려 결과창이 화면의 중간에 오도록하여 피드백을 줄 예정이다. 또한 가장 큰문제인 마커 말풍선과 내위치 기능인데 가능하면 구글api로 계속 유지하고 싶지만 정보량이 파씽한 공공api로 부족하다고 판단이 되면 네이버,다음 지도api로 바꿔 직접 주소를 입력해서 정보를 가져오는 방법도 고려하고 있다 또한 내 위치 기능은 실행을 해보았지만 오차범위가 너무 커서 그 문제를 찾고 해결이 될 때 추가해줄 예정이다.
